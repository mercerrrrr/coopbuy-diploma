# Implementation Alignment

## Ключевые проектные решения

- **Аутентификация** — по `email + password`; сессия хранится в httpOnly JWT-куке `cb_session` (7 дней).
- **Поставщик** — внешний контрагент; управляется администратором через `/admin/suppliers`.
- **Роли**: `ADMIN`, `OPERATOR`, `RESIDENT`; роль `OPERATOR` ограничена своим пунктом выдачи.
- **Функции организатора** распределены между `ADMIN` (справочники, пользователи, отчёты, настройки) и `OPERATOR` (выдача, приёмка, оплаты).
- **Масштабируемость** — сущности `Region`, `Settlement`, `PickupPoint`, `SupplierDeliveryZone` поддерживают несколько регионов; текущий пилотный сценарий работает в одном регионе.

## Реально реализованные роли

- `ADMIN` — управляет справочниками, локациями, пользователями, поставщиками, закупками, отчётами и оплатами.
- `OPERATOR` — работает в пределах своего пункта выдачи: видит связанные закупки, ведёт выдачу и приёмку.
- `RESIDENT` — житель населённого пункта, оформляет заявки, отслеживает свои заказы и уведомления.

## Предметная модель текущей реализации

- Поставщик реализован как внешний контрагент, от которого поступают товары и прайс-листы.
- Аутентификация реализована по `email + password`.
- Телефон хранится как контактное поле пользователя и участника заявки, но не используется как способ входа.
- Пилотный сценарий ориентирован на один региональный контур эксплуатации, при этом основным пользовательским выбором выступает `Settlement`.
- Сущности `Region`, `Settlement`, `PickupPoint`, `SupplierDeliveryZone` позволяют масштабировать систему на несколько территорий без смены архитектуры.
- Функции условного “организатора закупки” в текущем проекте распределены между ролями `ADMIN` и `OPERATOR`.

## Доступ к `/p/[code]`

- Страница закупки по invite-коду доступна для просмотра без авторизации.
- Любые действия участия (`addToCart`, `submitOrder`, редактирование корзины) доступны только авторизованному пользователю с ролью `RESIDENT`.
- Для участия `session.settlementId` жителя должен совпадать с `procurement.settlementId`.
- При несовпадении населённого пункта закупка остаётся доступной для просмотра, но участие блокируется с понятным сообщением.

## Жизненный цикл закупки

- Активной считается закупка со статусом `OPEN`, у которой не истёк дедлайн.
- После дедлайна закупка автоматически переводится в `CLOSED`.
- Если после дедлайна `submittedTotal < minTotalSum`, интерфейс явно показывает, что минимальная сумма не достигнута и закупка закрыта без продолжения сценария.
- Форма создания закупки визуально разделена на основные поля и дополнительные настройки, чтобы пилотный сценарий был проще для администратора.

## Операционные детали

- Выдача заказов выполняется по QR/ID заказа через `PickupSession` и `PickupCheckin`.
- Оплата фиксируется на уровне заказа с расчётом `goodsTotal`, `deliveryShare`, `grandTotal`.
- Страница `/my/orders/[orderId]` является основной пользовательской квитанцией; PDF-маршрут сохранён как экспортный вариант.
- Экспорты PDF/XLSX/CSV и журнал аудита используют единые подписи статусов и согласованную идентификацию сущностей.
