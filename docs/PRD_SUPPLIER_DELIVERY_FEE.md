# PRD: Стоимость доставки на уровне поставщика

## Проблема

Сейчас `deliveryFee` (общая стоимость доставки) задаётся вручную администратором при создании закупки. Если админ не указал значение — поле остаётся 0, участники не видят приблизительную стоимость доставки, и это создаёт путаницу.

## Решение

Перенести стоимость доставки на уровень **поставщика**. При создании закупки `deliveryFee` подтягивается автоматически из выбранного поставщика.

## Изменения

### 1. Схема данных (Prisma)

- **Supplier**: добавить поле `deliveryFee Int @default(0)` — базовая стоимость доставки поставщика.
- **Procurement.deliveryFee** — остаётся как есть. При создании закупки копируется из `supplier.deliveryFee`. Админ может переопределить при необходимости.

### 2. Админ-панель: Поставщики

- **Форма создания поставщика** (`createSupplier`): добавить поле «Стоимость доставки, ₽» (число ≥ 0).
- **Страница поставщика**: показывать текущую стоимость доставки в карточке.

### 3. Админ-панель: Создание закупки

- **Форма создания закупки** (`createProcurement`):
  - При выборе поставщика — автозаполнить поле `deliveryFee` значением из `supplier.deliveryFee`.
  - Поле остаётся редактируемым (админ может изменить для конкретной закупки).
  - Подсказка: «Подтянуто из карточки поставщика. Можно изменить.»

### 4. Публичная страница закупки (`/p/[code]`)

- В боковой панели (корзина) — **всегда** показывать блок приблизительной доставки, если `deliveryFee > 0`:
  - **До оформления** (корзина): amber-блок «≈ Доставка: ~X ₽ (пересчитывается при каждом новом участнике)».
  - Формула: `deliveryFee / (submittedCount + 1)` — текущий пользователь считается потенциальным участником.
  - Показывать даже если корзина пуста (чтобы участник видел ориентир до добавления товаров).
  - **После оформления** (submitted): показывать актуальную долю из `order.deliveryShare`.

### 5. Seed

- Добавить `deliveryFee` к существующим поставщикам в seed (АгроОпт-Юг: 1200, СберПоставка: 600).
- Добавить населённый пункт «Новолесное» в Астраханскую область.
- Добавить новую OPEN закупку в Новолесном с нормальными данными для тестирования.

## Что НЕ меняется

- `recalcDeliveryShares()` — логика пересчёта при submit остаётся прежней.
- `deliverySplitMode` — режим деления остаётся на закупке.
- Все существующие закупки с ручным deliveryFee продолжают работать.

## Миграция

- `ALTER TABLE "Supplier" ADD COLUMN "deliveryFee" INTEGER NOT NULL DEFAULT 0;`
- Данные: проставить текущие fee из активных закупок обратно в поставщиков (опционально, можно вручную).
