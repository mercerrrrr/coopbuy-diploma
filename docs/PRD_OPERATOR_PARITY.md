# PRD: Оператор = Админ своего пункта (Operator Parity)

## Статус: Draft
## Дата: 2026-04-14

---

## Проблема

Оператор привязан к конкретному пункту выдачи (`pickupPointId`), но имеет
существенно урезанные права по сравнению с админом. Это создаёт несостыковки
в бизнес-логике:

- Оператор **создаёт закупки**, но **не может импортировать прайс-лист** поставщика
- Оператор **принимает товар физически**, но **не может оформить акт приёмки**
- Оператор **выдаёт заказы по QR**, но **не может управлять оплатой и возвратами**
- Оператор **не может настроить доставку** для своей закупки

## Принцип

> **Оператор = Администратор, ограниченный своим пунктом выдачи.**

Оператор должен иметь **весь функционал** админа, кроме:
- Управления пользователями (создание операторов/жителей)
- Управления локациями (регионы, НП, пункты выдачи)
- Управления поставщиками (CRUD: создание, удаление, вкл/выкл, зоны доставки)

Эти разделы остаются **только у ADMIN**, т.к. они затрагивают глобальную
структуру системы. Поставщик — глобальная сущность, один поставщик может
обслуживать несколько НП/ПВЗ. Оператор **видит** доступных поставщиков
и **импортирует прайс-листы**, но не создаёт/удаляет самих поставщиков.

## Текущее состояние доступа

### Оператор МОЖЕТ (уже работает)
- Просмотр дашборда (свой пункт)
- Список и создание закупок (свой пункт)
- Закрытие закупок
- Создание сессий выдачи, чекин заказов
- Экспорт документов (XLSX/CSV/PDF)

### Оператор НЕ МОЖЕТ (нужно открыть)
| Функция | Текущий guard | Нужен guard |
|---|---|---|
| Просмотр страницы поставщика | `assertAdmin()` на page.jsx | `assertOperatorOrAdmin()` (read-only для оператора) |
| Импорт прайс-листа (draft/apply) | `getSession()` без проверки роли* | `assertOperatorOrAdmin()` |
| Справочники: категории, ед. изм. | `assertAdmin()` | `assertOperatorOrAdmin()` |
| Настройка доставки | `requireAdminOnlyProcurementAction()` | `requireProcurementForAdminAction()` |
| Пересчёт долей доставки | `requireAdminOnlyProcurementAction()` | `requireProcurementForAdminAction()` |
| Управление статусом оплаты | `requireAdminOnlyProcurementAction()` | `requireProcurementForAdminAction()` |
| Возврат платежа (refund) | `requireAdminOnlyProcurementAction()` | `requireProcurementForAdminAction()` |
| Создание акта приёмки | `requireAdminOnlyProcurementAction()` | `requireProcurementForAdminAction()` |
| Редактирование строк приёмки | `requireAdminOnlyProcurementAction()` | `requireProcurementForAdminAction()` |
| Финализация акта приёмки | `requireAdminOnlyProcurementAction()` | `requireProcurementForAdminAction()` |

\* Импорт прайс-листа в `src/app/admin/suppliers/[id]/import/actions.js` проверяет `getSession()`, но не проверяет роль — формально баг безопасности, который тоже закроем.

### Оператор НЕ ДОЛЖЕН мочь (остаётся ADMIN only)
| Функция | Причина |
|---|---|
| Управление пользователями | Глобальная операция, не привязана к пункту |
| Управление локациями (регионы, НП, ПВ) | Структурные данные всей системы |
| Поставщики: CRUD (создание, удаление, вкл/выкл, зоны доставки) | Глобальная сущность — один поставщик обслуживает несколько ПВЗ |

## Фаза 4 — Контекст ПВЗ в интерфейсе

### Проблема

Сейчас ни оператор, ни житель не видят в шапке свою привязку:
- Оператор видит только «Рабочий кабинет · Оператор · email» — **не видит** название
  и адрес своего ПВЗ
- Житель видит только «Личный кабинет · Житель · ФИО» — **не видит** свой НП
- Житель видит название ПВЗ в карточке закупки, но **не видит адрес** ПВЗ до
  оформления заказа — непонятно куда ехать забирать

### Что нужно сделать

#### 4.1 Шапка admin layout — показать ПВЗ оператора

**Файл:** `src/app/admin/layout.jsx`

- Для OPERATOR: подгрузить `pickupPoint` (name + address) из БД по `session.pickupPointId`
- Показать под email: **«ПВЗ: {name} · {address}»**
- Для ADMIN: показать **«Все пункты»** (или ничего — у админа нет привязки)

#### 4.2 Шапка my layout — показать НП жителя

**Файл:** `src/app/my/layout.jsx`

- Подгрузить `settlement` (name) + `settlement.region` (name) из БД по `session.settlementId`
- Показать под ФИО: **«{region.name} · {settlement.name}»**
- Если `settlementId` нет — показать подсказку «НП не указан»

#### 4.3 Информация о ПВЗ для жителя в карточке закупки

**Файл:** `src/app/my/procurements/page.jsx`

Сейчас в карточке закупки уже есть `procurement.pickupPoint.name`.
Нужно добавить **адрес**: `procurement.pickupPoint.address`.

Показать как:
```
📍 ПВЗ «Центральный» · ул. Ленина, 42
```

Аналогично в:
- `src/app/my/orders/page.jsx` — список заказов
- `src/app/my/orders/[orderId]/page.jsx` — детали заказа (тут уже есть в receipt.pdf,
  но не на самой странице)
- `src/app/p/[code]/page.jsx` — публичная страница закупки (до оформления заказа
  житель должен видеть куда приехать)

#### 4.4 Выбор ПВЗ при регистрации (опционально для MVP)

Сейчас при регистрации житель выбирает только НП (`settlementId`).
В одном НП может быть несколько ПВЗ — житель не выбирает предпочитаемый.

**Это НЕ критично**, т.к. ПВЗ привязан к закупке, а не к жителю. Житель
видит все закупки своего НП и сам выбирает, в какой участвовать. Но для
полноты можно добавить каскадный выбор НП → ПВЗ на странице регистрации
(или в профиле), чтобы фильтровать закупки по предпочитаемому ПВЗ.

**Решение для MVP:** не добавлять выбор ПВЗ при регистрации. Достаточно
показывать адрес ПВЗ в карточках закупок — житель сам увидит, какой пункт
ему ближе.

## Риски

- **Минимальный**: все данные уже фильтруются по `pickupPointId` для оператора
  через `requireAccessibleProcurement()` / `requireProcurementForAdminAction()`.
  Открытие доступа не даёт оператору видеть чужие закупки.
- **Поставщики** — глобальная сущность. CRUD остаётся у ADMIN. Оператор
  видит список (read-only) и импортирует прайс-листы — это безопасно.
- **Справочники** (категории, ед. изм.) — тоже глобальные, но оператору
  нужен доступ при импорте прайса (создать недостающую категорию).
  Допустимо для MVP.
- **Адрес ПВЗ** — поле `address` уже обязательное в схеме (`String`, не optional),
  так что данные гарантированно есть.
